📑 Document Outline
① The Problem It Solves

The README documents device commands like /shot and /tap, but it leaves out the strongest path: turning Telegram into a "remote control for your AI coding assistant." You're out, your Mac is at home, and iTerm2 already has Claude Code / Codex sessions running — now you can hand them tasks from your phone and get the AI's answers back.

❌ Without this path

You must sit at the machine, switch to the right iTerm tab, type to Claude Code by hand, and wait at the desk to see results. Leave your seat and you're disconnected.

✅ With this path

Send "[t3] fix the login bug" from your phone → it lands in the Claude Code in tab 3 → the AI works → the reply is pushed back to Telegram. Keep an eye on several projects even on your commute.

💡
Core idea: this isn't "remote SSH" — it's a two-way bridge. Telegram text goes in, the AI assistant's reply comes out, with iTerm2's AppleScript injection and screen-buffer capture connecting the two ends.
② Architecture Diagram

Three stages: inbound tg-relay → term-bridge inject/capture → outbound tg-notify, with the actual working iTerm2 tab in the middle.

③ Life of a Message

From the moment you hit send until the AI reply pops back to your phone, it passes through 5 stages:

1
Send + parse prefix
tg-relay receives [t3] fix login bug; parse_routed_message() recognizes the prefix [t3] → target w1/t3, and strips the body down to "fix login bug".
2
Inject into the iTerm2 tab
iterm-inject.py uses AppleScript to locate the active session of window 1 / tab 3 and write text the body plus Enter — like "typing into that tab on your behalf." It also backs up to inbox/pending.txt.
3
Claude Code / Codex executes
The AI running in that tab takes the line as a prompt and starts reading code, editing, running commands. Output is also written to the local log buffer inbox/iterm-session-*.log so long replies aren't cut off by iTerm's default line limit.
4
Capture after a delay
TG_ITERM_MONITOR_AFTER seconds after injection (default ~45s), iterm-monitor starts polling the screen buffer and uses extract_latest_reply() to filter out prompts and pull out just the assistant's answer.
5
Return to Telegram
Once it detects the reply is "done," it pushes it back to your private chat via tg-notify send. If there's no new output for a long time, it can fall back to sending an iTerm screenshot.

④ Multi-Tab Routing: Who Gets the Message

You might have several tabs open, each for a different project, each running its own AI session. A message prefix decides which tab the line goes to — switch back and forth within one conversation, no config changes needed.

MethodSyntaxExampleRule
By tab number[tN] / #N / @tN:[t3] list dirMost common, N = tab index
By directory/project name[name] / @name:[myapp] run testsFuzzy match against the directory fragment in the tab title
By .env alias[alias][fz] check deployExact mapping, most reliable when dir names collide
No prefix(just the body)check git statusFalls to the .env default tab
# .env —— exact aliases for duplicate dir names (alias:window:tab)
TG_ITERM_ALIASES=fz:1:7,myapp:1:11,api:1:6

Routing decision (logic inside iterm_route.py):

no prefix
→ use the .env default tab from TG_ITERM_WINDOW / TG_ITERM_TAB
prefix is a number
→ locate by tab index (when the same index is ambiguous, prefer window 1)
prefix is alias/name
→ check TG_ITERM_ALIASES first, then fuzzy-match the title; with multiple candidates, take the shortest name (most specific)
🧭
Send /tabs from your phone and the bot lists all current iTerm tabs with suggested prefixes — no need to memorize indices.
⑤ Return & Completion (the trickiest part)

The hard part: terminal output is streaming — how do you know the AI is "done" and which segment to send back? iterm-monitor.py judges with three signals:

① Completion marker

is_reply_complete() recognizes the signs of Claude Code / Codex finishing an answer (prompt returns, end markers); a hit means it's done writing.

② Stable polling

If the captured content stops changing for TG_ITERM_MONITOR_STABLE_POLLS consecutive polls (default 2), the output is considered stable and ready to send.

③ Dedup

If the extracted result equals "the last sent" one (the last-sent state file), it's skipped — avoiding the same reply flooding the chat.

④ Screenshot fallback

If for more than TG_ITERM_MONITOR_SCREENSHOT_IDLE seconds (default 60) there's neither new text nor an extractable reply, it sends an iTerm screenshot instead.

⚠️
Only the assistant reply, no noise: extract_latest_reply() filters out shell prompts, the text you injected, progress bars, etc., aiming to return only the AI's actual answer. Each tab has its own state files inbox/iterm-monitor-w1-t{N}.* so they never cross-talk.
🔒
Return goes to a private chat: iterm-monitor requires TELEGRAM_CHAT_ID to be a positive number (private chat); a group (negative) is rejected — to avoid leaking code/output into a group.
⑥ Quick Start

Prerequisites: Bot Token and Chat ID configured per docs/TELEGRAM_SETUP.md, and an iTerm2 tab already open running Claude Code / Codex.

1
Configure .env to enable injection
TG_RELAY_ITERM_INJECT=1 (natural language → injection), TG_ITERM_MONITOR_AFTER=45 (how long after injection to start capturing the return).
2
Increase the iTerm scrollback (once)
./mob iterm-buffer-setup — avoids long replies being cut off by the default 1000-line limit.
3
See which tabs you have
./mob iterm-list — lists each tab's index, title, and suggested prefix/alias.
4
Start the full stack
./mob up — launches both tg-relay (receive) and iterm-monitor (return output).
5
Send from your phone
In Telegram send [t3] refactor this function for me; about 45s later Claude Code's answer comes back to your phone.
# Local CLI cheat sheet
./mob up # start relay + monitor
./mob down # stop
./mob iterm-list # list tabs
./mob iterm-route "[myapp] test" # test route parsing locally
✅
One line to remember it: Telegram is the remote control, the Claude Code / Codex in the iTerm2 tab is the robot arm, and term-bridge is the cable connecting them — text goes in, the AI's reply comes out.
