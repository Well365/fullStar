用 Telegram 远程驱动 iTerm2 里的 Claude Code / Codex

在手机上给 Telegram Bot 发一句话 → 自动注入到 Mac 上某个正在跑 Claude Code / Codex 的 iTerm2 标签页 → AI 执行 → 助手回复自动回传到 Telegram。 人不在电脑前，也能隔空指挥终端里的 AI 编程助手。

tg-relay 入站 term-bridge 注入/捕获 tg-notify 出站 macOS · iTerm2 · AppleScript
📑 文档大纲
① 它解决什么问题

README 里写了 /shot、/tap 这类设备控制命令，但没有写出最强的那条链路： 把 Telegram 当成「AI 编程助手的遥控器」。你人在外面，Mac 留在家里，iTerm2 里早就开好了 Claude Code / Codex 会话——现在可以用手机直接给它派活，并收到 AI 的回答。

❌ 没有这条链路时

必须坐在电脑前，手动切到对应 iTerm 标签页，敲字给 Claude Code，等它跑完再看结果。离开座位就断联。

✅ 有了这条链路

手机发「[t3] 修一下登录的 bug」→ 注入 tab 3 的 Claude Code → AI 干活 → 回复自动推回 Telegram。通勤路上也能盯着多个项目。

💡
核心定位：它不是「远程 SSH」，而是一座双向桥—— Telegram 文字进、AI 助手回复出，中间靠 iTerm2 的 AppleScript 注入与屏幕缓冲捕获打通。
② 整体架构示意图

三段式：入站 tg-relay → term-bridge 注入/捕获 → 出站 tg-notify。中间夹着真正干活的 iTerm2 标签页。

角色组件 / 文件职责
入站tg-relay/ · iterm_route.py轮询 Telegram Bot，解析消息前缀，决定路由到哪个 tab，剥掉前缀只留正文
注入term-bridge/iterm-inject.py用 osascript AppleScript 向目标 window/tab/session 模拟键入正文并回车
执行iTerm2 标签页内的 Claude Code / Codex把注入的文字当作 prompt，真正去读代码、改文件、跑命令
捕获iterm-monitor.py · iterm_extract.py轮询屏幕缓冲，提取「助手最新回复」，判断是否回答完毕
出站tg-notify send把提取到的回复发回你的 Telegram 私聊（同一个 Bot）
③ 一条消息的生命周期

从你按下发送，到 AI 回复弹回手机，完整经过 5 个阶段：

1
发送 + 解析前缀
tg-relay 收到 [t3] 修登录 bug，parse_routed_message() 识别前缀 [t3] → 目标 w1/t3，正文剥成「修登录 bug」。
2
注入 iTerm2 标签页
iterm-inject.py 通过 AppleScript 定位 window 1 / tab 3 的活动 session，write text 键入正文并回车——相当于「替你在那个 tab 里打字」。同时备份到 inbox/pending.txt。
3
Claude Code / Codex 执行
该 tab 里跑着的 AI 助手把这句话当 prompt，开始读代码、编辑、跑命令。输出同时写进本地日志缓冲 inbox/iterm-session-*.log，避免长回复被 iTerm 默认行数截断。
4
延迟后开始捕获
注入 TG_ITERM_MONITOR_AFTER 秒（默认约 45s）后，iterm-monitor 开始轮询屏幕缓冲，用 extract_latest_reply() 滤掉提示符、只抠出助手的那段回答。
5
回传 Telegram
检测到回复「写完了」就用 tg-notify send 推回你的私聊。长时间无新输出时，可改发一张 iTerm 截图兜底。
# 手机里看到的一来一回
你 ▸ [t3] 修一下登录接口 500 的问题
bot ▸ typed into iTerm tab 3 (../myapp (-zsh)) ; 已注入
……（约 45s 后）……
bot ▸ 已定位：login.py 里 token 解析未捕获 KeyError，
已加 try/except 并补了测试，3 个用例通过 ✅
④ 多 Tab 路由：消息发给谁

你可能同时开了好几个 tab，每个对应不同项目、各自跑着一个 AI 会话。消息前缀决定这句话进哪个 tab，无需改配置即可在一段对话里来回切换。

方式写法示例规则
按标签序号[tN] / #N / @tN:[t3] 列目录最常用，N = tab 序号
按目录/项目名[名字] / @名字:[myapp] 跑测试模糊匹配 tab 标题里的目录片段
按 .env 别名[别名][fz] 看部署精确映射，目录重名时最稳
无前缀（直接写正文）看下 git 状态落到 .env 默认 tab
# .env —— 给重名目录起精确别名（别名:窗口号:标签号）
TG_ITERM_ALIASES=fz:1:7,myapp:1:11,api:1:6

路由决策（iterm_route.py 内部逻辑）：

无前缀
→ 用 .env 的 TG_ITERM_WINDOW / TG_ITERM_TAB 默认 tab
前缀是数字
→ 按 tab 序号定位（同序号有歧义时优先 window 1）
前缀是别名/名字
→ 先查 TG_ITERM_ALIASES，没命中再模糊匹配标题，多个候选取最短名（最具体）
🧭
手机发 /tabs 即可让 Bot 列出当前所有 iTerm 标签页和推荐前缀，不用记序号。
⑤ 回传与完成判定（最巧的一环）

难点在于：终端输出是流式的，怎么知道 AI「说完了」、该把哪一段发回去？iterm-monitor.py 用三重信号判断：

① 完成标记

is_reply_complete() 识别 Claude Code / Codex 回答收尾的特征（提示符回归、结束符），命中即视为写完。

② 稳定轮询

连续 TG_ITERM_MONITOR_STABLE_POLLS 次（默认 2）抓到的内容不再变化，就认为输出稳定、可以发送。

③ 去重

提取结果与「上次已发」相同则跳过（last-sent 状态文件），避免同一段回复重复刷屏。

④ 截图兜底

若超过 TG_ITERM_MONITOR_SCREENSHOT_IDLE 秒（默认 60）既没新文字也提取不到，就改发一张 iTerm 截图。

⚠️
只发助手回复，不发噪音：extract_latest_reply() 会滤掉 shell 提示符、你注入的原文、进度条等，尽量只回传 AI 真正的那段回答。每个 tab 有独立状态文件 inbox/iterm-monitor-w1-t{N}.*，互不串台。
🔒
回传走私聊：iterm-monitor 要求 TELEGRAM_CHAT_ID 是正数（私聊），群聊（负数）会被拒——避免把代码/输出误发进群。
⑥ 快速开始

前置：已按 docs/TELEGRAM_SETUP.md 配好 Bot Token 与 Chat ID，且 iTerm2 里已开好跑着 Claude Code / Codex 的标签页。

1
配置 .env 打开注入
TG_RELAY_ITERM_INJECT=1（自然语言→注入），TG_ITERM_MONITOR_AFTER=45（注入后多久开始抓取回传）。
2
增大 iTerm 滚动缓冲（一次）
./mob iterm-buffer-setup —— 避免长回复被默认 1000 行限制截断。
3
查看有哪些 tab
./mob iterm-list —— 列出每个 tab 的序号、标题和推荐前缀/别名。
4
启动全套服务
./mob up —— 同时拉起 tg-relay（收消息）和 iterm-monitor（回传输出）。
5
手机发消息
在 Telegram 发 [t3] 帮我重构这个函数，约 45s 后 Claude Code 的回答就回到手机上。
# 本地 CLI 速查
./mob up # 启动 relay + monitor
./mob down # 停止
./mob iterm-list # 列出 tab
./mob iterm-route "[myapp] test" # 本地测试路由解析
✅
一句话记住它：Telegram 是遥控器，iTerm2 标签页里的 Claude Code / Codex 是机械臂， term-bridge 是连接两者的线缆——发文字进去，收 AI 回复出来。
